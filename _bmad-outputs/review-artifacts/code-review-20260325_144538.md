## Review Findings

- `controller/agent/nodes/dof_guard.py` stamps the canonical DOF checklist key as `fail` for every non-`APPROVED` review, and both engineering reviewer nodes apply that helper before persisting the result: [dof_guard.py](/home/maksym/Work/proj/Problemologist/Problemologist-AI/controller/agent/nodes/dof_guard.py#L120-L133), [plan_reviewer.py](/home/maksym/Work/proj/Problemologist/Problemologist-AI/controller/agent/nodes/plan_reviewer.py#L155-L165), [execution_reviewer.py](/home/maksym/Work/proj/Problemologist/Problemologist-AI/controller/agent/nodes/execution_reviewer.py#L397-L401).
  - This conflates DOF-specific failure with unrelated reviewer rejections. A plan or execution review rejected for budget, render-evidence, simulation, or other non-DOF reasons will still be recorded as `dof_minimality: fail` or `dof_deviation_justified: fail`, which corrupts the telemetry this story is trying to preserve.
  - That breaks the intent of the story and the architecture contract: the checklist should distinguish minimality from justified deviation, not merely mirror the overall decision.
  - Fix: only stamp the canonical DOF key from the deterministic DOF branch, or preserve a reviewer-provided `not_applicable`/existing DOF value when the rejection reason is unrelated to DOF.
