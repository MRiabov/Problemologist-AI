### Inputs to the system

The inputs to each model are as described in @desired_architecture.md document. What this means for dataset generation is that the system must supply and persist the dataset in and out from the system. E.g., a benchmark coder would have in each dataset row (quoted verbatim from desired architecture):

```md
- plan.md (read)
- todo.md (read/write)
- benchmark_definition.yaml (read)
- assembly_definition.yaml (read)
- reviews/** (read)
- renders/** (read, tool-generated)
- journal.md (read/write)
- benchmark_script.py / solution_script.py / template-backed implementation file(s) (role-appropriate read/write; `benchmark_script.py` appears only after benchmark implementation exists; plus additional *.py)
- plan_refusal.md (write only, when refusing plan)
```

Note that skills and utils files are loaded at runtime (currently from the most recent version, to avoid git work)

When the exact same starter artifact recurs across rows or agent families, store that body once under `shared/agent_templates/` and copy it into the workspace from there. Dataset rows should keep only row-specific
custom artifacts in their seed folders; common boilerplate such as `todo.md` and `journal.md` should not be duplicated
in the row bundle. The authored `benchmark_script.py` and `solution_script.py` files remain row-specific artifacts once their respective stages exist.

This means that the dataset rows (it actually is archives on s3; the only item in DB is metadata) must contain all of them; or nulls where nullable. They will be later persisted as files.

Dev note: to clarify: we already have the actual files in s3; no point in duplicating it in the DB; plumb in and out. The DB will only store metadata, and we'll be able to filter through it.

## Cleaning

We hard filter out the following data:

1. Integration test runs: these should be recorded at runtime somewhere. Ideally, the integration tests should be during tests persisted into a different DB to avoid pollution at source (they are already during IS_INTEGRATION_TEST=true)
2. Failed data due to environment or LLM failure; this shouldn't normally happen because we already have hard pass gates.

### Time-based cleaning

We know that all data before march 3 Dublin time 00:00 (midnight) was corrupted (buggy environment), which means it is unusable for our purpose, exclude it from the dataset.

### Dataset based cleaning

We can filter out interesting data based on dataset events of that dataset. If that jsonl isn't available as a file, it should be in the database (I think it is available in the dataset).

## Actually generating the data

The data must be actually batch generated by running some script. Technically, data/run_evals.py is already suitable as a batch generator, however it also includes logic of dataset generation.

The dataset generator script has the following requirements:

1. Dataset generation:
   - The script will load data from the previous seed dataset, prompt the models to run (through endpoints of the controller app in @desired_architecture.md) and persist the data into @desired_architecture.md
   - The script will not blindly iterate through all seeds, instead, it will prioritize seeds that are underrepresented in data
   - The script has a switch to not generate from seeds that have already been generated from.
   - The script will record from which seed the data was generated. (how? not sure exactly. Probably via metadata. That said; we already have some hack-ish logic for integration tests to record their IDs as UUID5 which is deterministic; as far as I know, the UUIDs are generated from INT-xxx numbers of integration tests.
