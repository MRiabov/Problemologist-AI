# DSPy ReAct dependency research

## Scope summary

- Primary focus: clarify what DSPy `ReAct` actually does in this repository environment.
- Goal: prevent architecture drift and repeated arguments about whether `next_*` fields are project code or DSPy-internal behavior.
- This document is evidence-only and refers to the installed dependency source code, not assumptions.

## Why this document exists

`specs/architecture/agents/tools.md` intentionally describes the desired tool surface at a high level (`tools=[func_a, func_b]`).
That high-level API statement is correct.

The confusion came from a second question: what protocol does the currently installed `dspy.ReAct` implementation use internally to run the loop?

## Environment verified in this repo

- Python environment: `.venv`
- DSPy package version: `3.1.3`
- Source inspected: `.venv/lib/python3.12/site-packages/dspy/predict/react.py`

## Verified facts from dependency source

### 1. Passing plain functions is supported

In `dspy.ReAct.__init__`, tools are accepted as callables and wrapped:

- `tools = [t if isinstance(t, Tool) else Tool(t) for t in tools]`

This confirms that using plain Python functions in `tools=[...]` is valid.

### 2. The internal loop contract in this version is `next_*`

In the same file, the instruction text generated by DSPy explicitly says:

- `interleave next_thought, next_tool_name, and next_tool_args`

DSPy also builds a signature that appends output fields:

- `next_thought`
- `next_tool_name`
- `next_tool_args`

At execution time, `forward()` consumes exactly these fields:

- `pred.next_thought`
- `pred.next_tool_name`
- `pred.next_tool_args`

This proves that for DSPy `3.1.3`, `next_*` is not invented by our app; it is how this dependency version implements ReAct.

### 3. Adapter choice does not remove the `next_*` contract for this class

`ReAct` uses `dspy.Predict(react_signature)` where `react_signature` already contains `next_*` outputs.

`ChatAdapter` appears in `_format_trajectory` as fallback formatting logic for trajectory rendering:

- `adapter = dspy.settings.adapter or dspy.ChatAdapter()`

This does not rewrite the action field schema from `next_*` to another protocol.

## Reconciliation with `specs/architecture/agents/tools.md`

Both of the following statements are true at the same time:

1. `tools=[python functions]` is correct and supported.
2. In DSPy `3.1.3`, the built-in `ReAct` class still uses an internal `next_*` loop schema.

Therefore, when discussing architecture:

- The **tool surface contract** should stay function-based and human-readable (`write_file`, `read_file`, etc.).
- We should avoid leaking dependency-internal field names (`next_tool_name`, etc.) into repository-level architecture contracts and test semantics.

## Architectural guidance for implementation agents

If a task requires strict "native tool call" behavior without dependency-specific `next_*` fields appearing in app-level contracts, do not rely on assumptions from high-level snippets alone.

Required implementation discipline:

1. Verify the installed dependency behavior from source in `.venv`.
2. Keep dependency internals encapsulated behind runtime boundaries.
3. Keep repository specs and integration assertions focused on observable tool behavior (`TOOL_START/TOOL_END`, tool names, arguments, outcomes), not dependency-private field names.
4. Fail closed if bridging logic starts inferring success from text shims or fallback parsing.

## Non-goals of this document

- This document does not prescribe a migration strategy.
- This document does not claim that all DSPy versions behave identically.
- This document is a point-in-time dependency audit for this repository environment.
