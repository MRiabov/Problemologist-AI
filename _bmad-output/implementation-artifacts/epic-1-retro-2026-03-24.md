# Epic 1 Retrospective

Date: 2026-03-24
Epic: 1 - Benchmark Creation & Validation

## Epic Review

Epic 1 landed the benchmark contract cleanly: geometry validation, explicit read-only benchmark ownership, fail-closed solvability checks, latest-revision handoff persistence, and reviewer-side solvability rejection are all now explicit and deterministic.

What went well:

- The core architecture already pointed in the right direction, so most work was contract alignment rather than new subsystem design.
- Story 1.1 and Story 1.2 tightened the benchmark seed schema and made ownership/interaction boundaries machine-readable.
- Story 1.3 and Story 1.5 converged on a single solvability vocabulary, which kept validation and reviewer decisions consistent.
- Story 1.4 made the latest revision the only active handoff surface and kept render evidence traceable.
- The implementation stories consistently relied on integration coverage, which matched the repo rule set and kept the validation path grounded in real workflows.

What was hard:

- Reviewer evidence had to stay aligned with the latest revision. Path or bundle drift could hide the real solvability reason until the inspected artifact path was normalized.
- The benchmark-to-engineer boundary is easy to blur in prose. The team had to keep the schema and validation layers explicit so benchmark-owned fixtures stayed read-only.
- The handoff path is sensitive to stale manifests and cross-revision artifacts. That made strict entry validation essential rather than optional.

Key lessons:

- Put ownership semantics in schema and validation, not prose.
- Reuse a single rejection vocabulary from validation through reviewer output.
- Treat visual inspection as a separate evidence step, not a substitute for file discovery.
- Keep latest revision, manifest scope, and session linkage explicit in every benchmark handoff path.

Technical debt carried forward:

- Any future benchmark template change now needs to preserve the interaction-permission, motion-contract, and latest-revision assumptions already established here.
- Render-path and manifest-path normalization need to stay protected, because a small mismatch can invalidate otherwise correct reviewer evidence.

## Next Epic Preparation

3. Keep `inspect_media(...)` mandatory whenever render evidence exists for a review path.
4. Maintain the explicit solvability reason mapping from validation into reviewer decisions.

Parallel preparation:

1. Seed engineer-side workflow coverage for approved benchmark intake and stale-revision rejection.
2. Keep regression coverage around manifest/session mismatches and render evidence traceability.
3. Keep story and template wording aligned with the read-only benchmark boundary so Epic 2 does not inherit ambiguity.

Nice-to-have preparation:

2. Keep a concise artifact-summary pattern for episode records so benchmark intake is easier to audit later.

Epic 2 does not require a scope change from Epic 1 findings. It does require the benchmark package contract to stay stable, because the engineering workflow builds directly on that handoff.
