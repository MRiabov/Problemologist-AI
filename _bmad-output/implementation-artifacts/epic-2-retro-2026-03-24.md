# Epic 2 Retrospective

Date: 2026-03-24
Epic: 2 - Human Solution Workflow

## Epic Review

Epic 2 completed the solution-side workflow cleanly. The team connected approved benchmarks to engineer runs, surfaced evidence and terminal metadata directly in the workspace, supported retry against the same benchmark without mutating the original episode, and added peer review under runtime jitter with fail-closed behavior.

What went well:

- The benchmark-to-engineer boundary stayed intact. Story 2.1 reused the approved benchmark package as read-only context instead of introducing a second storage path.
- Evidence selection became explicit. Story 2.2 moved the workspace toward latest solution evidence first, with terminal metadata rendered as first-class data rather than inferred from generic failure state.
- Retry stayed lineage-safe. Story 2.3 preserved `benchmark_id`, tracked `prior_episode_id`, and kept the revision chain visible without rewriting the failed run.
- Peer stability review remained grounded in persisted evidence. Story 2.4 kept review decisions tied to jitter results and render inspection instead of optimistic status labels.
- Integration coverage matched the repository contract. The work leaned on live integration validation and deterministic seeded cases where appropriate, which kept the implementation aligned with real workflow behavior.

What was hard:

- The same contract had to be enforced in several places. Latest-revision rules, immutable benchmark ownership, and evidence selection surfaced in multiple stories, so the work required careful consistency across controller, frontend, and review paths.
- Workspace affordances needed stable hooks. Hidden debug payloads and explicit test IDs were necessary to make the complex evidence and retry flows testable without brittle DOM scraping.
- Some validation runs were noisy because the broader integration environment can fail for unrelated reasons. That made it important to keep story-specific scenarios narrow and deterministic where possible.

Key lessons:

- Make lineage explicit in persisted metadata instead of reconstructing it from UI state or free-form text.
- Prefer the latest persisted evidence artifact over plan files when reviewing completed engineer runs.
- Keep retry behavior additive and immutable. A new attempt should create a new engineer episode, not rewrite the previous one.
- Use the existing controller bootstrap and review handover paths as the source of truth. Avoid parallel episode stores or custom retry transport.
- Keep render inspection and review decisions fail closed. If evidence is missing, the system should surface absence instead of synthesizing success.

## Next Epic Preparation

Epic 3 adds cost, weight, and manufacturability checks. The main preparation work is to keep the Epic 2 solution workflow stable while those additional constraints are layered on.

Critical preparation:

1. Preserve the immutable benchmark handoff contract so cost and manufacturability checks only operate on engineer-owned solution artifacts.
2. Keep terminal metadata and evidence selection available for reviews, because Epic 3 will depend on the same inspectable run history.
3. Retain lineage fields such as `benchmark_id`, `prior_episode_id`, and reuse state so revised solutions remain comparable against the same benchmark.
4. Keep review and retry controls fail closed for any run that lacks the required persisted evidence.

Preparation tasks:

1. Reuse the existing solution-evidence selection pattern when surfacing manufacturability and cost evidence.
2. Keep integration fixtures deterministic for Epic 3 so quantity-based and COTS-related validation stays reproducible.
3. Preserve explicit selectors and debug hooks for workspace evidence views instead of falling back to brittle text matching.

Technical debt to carry forward:

- The hidden debug payload fields used by the workspace tests should be consolidated into a reusable pattern if future stories keep adding more evidence surfaces.
- The integration suite would benefit from more tightly scoped scenarios for story-level validation so unrelated harness regressions are easier to distinguish from story defects.
- Review and evidence-selection logic is now duplicated across several screens and should stay synchronized as new solution evidence types appear.
